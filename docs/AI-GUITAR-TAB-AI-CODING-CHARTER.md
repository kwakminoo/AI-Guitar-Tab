# AI-Guitar-Tab — AI 코딩 가이드 (범위·목표·용어 고정)

이 문서는 **Cursor 등 AI 도구로 코드를 수정·추가할 때** 방향이 요청이나 프로젝트 목표에서 벗어나지 않도록 하는 **단일 기준 문서**다.  
AI/개발자는 **이 파일과 `README.md`의 파이프라인 요약**을 우선한다.

---

## 1. 프로젝트 한 줄 목표

**유튜브(또는 동영상) 링크**를 넣으면 오디오를 내려받고, **스템 분리** 후 **기타(없으면 피아노) 트랙**에서 **MIDI**를 만들고, 파이프라인으로 **AlphaTex / AlphaTab 스코어**를 생성해 **웹에서 악보를 표시**한다.

---

## 2. AI 코딩 시 반드시 지킬 것 (가드레일)

1. **사용자가 명시한 작업만** 한다. 추측으로 기능을 넓히지 않는다.
2. **이 문서의 용어·목표·비목표**를 위반하는 변경은 하지 않는다. 충돌 시 **사용자 확인**이 필요하다고 명시한다.
3. 변경 범위는 **요청된 파일·모듈**에 한정한다. “정리 차원”의 무관한 리팩터, 문서·설정 대량 수정은 **요청 없이 하지 않는다**.
4. 새 기능은 **백엔드 파이프라인(`backend/app/services/…`)** 또는 **프론트 뷰어(`frontend/src/…`)** 중 요청된 쪽에만 맞춘다.
5. **비밀·키·개인 경로**를 코드에 하드코딩하지 않는다.
6. ML/외부 바이너리 동작은 **환경 변수·기존 스크립트 패턴**을 따른다 (`README.md` 참고).

---

## 3. 핵심 파이프라인 (코드와 동일한 사실만)

`README.md` 기준 요약:

1. `yt-dlp` 등으로 오디오 다운로드  
2. **Demucs** 등으로 스템 분리  
3. 기타 스템 → **44.1kHz 모노 WAV** 등 전처리 후 **Basic Pitch** 등으로 MIDI  
4. 악보 **템포(BPM)** 는 **생성된 MIDI의 템포 이벤트**에서 사용  
5. MIDI·메타로 **AlphaTex** 및 프론트용 스코어 생성  

---

## 4. 용어 정리 (유비쿼터스 언어)

| 용어 | 정의 |
|------|------|
| **음원** | 스템 분리로 얻은 **기타(또는 대체 트랙)의 WAV/MP3 등** 재생 파일. 코드/문서에서 “스템 파일”과 혼용하지 말고 사용자 요청 시 **음원**을 쓴다. |
| **노드** | AlphaTab 악보에 표시되는 **연주 위치 정보**(프렛·줄 등). 예: `2.3`, 괄호로 묶인 `(x.6 x.5 2.2 2.3)` 형태의 표기 단위. |
| **온셋** | MIDI에서 **음이 시작되는 시각**. 타브의 “언제 치는지”의 기준. |
| **마디·바** | 악보의 박자 단위 구간. 온셋은 **마디/그리드** 위에 올려 표현한다. |

---

## 5. 제품 규칙 (도메인 결정 — 사용자 확정본)

아래는 **프로젝트가 지향하는 연주 가능한 타브**를 위한 규칙이다. 구현이 아직 완전히 일치하지 않을 수 있으나, **새 코드는 이 방향을 해치지 않도록** 한다.

1. **카포**  
   - 목적: **코드 진행을 단순·쉬운 형태로 맞추는 것**.  
   - 값: **0, 1, 2, 3, 4, 5** 중에서 곡 분석 결과에 맞게 선택. “항상 2프렛” 같은 고정값이 아니다.

2. **연주 가능성**  
   - 악보는 **사람이 연주할 수 있는 형태**를 목표로 한다.  
   - **코드가 있는 마디**에서는 해당 코드의 **주법(손가띨 위치 범위)** 안에서 음을 표현하는 것을 목표로 한다.  
     - 예: D 코드 사용 시 그 주법(예: `2 3 2 0 x x` 등) 안에서 최대한 표현.

3. **음정과 노드**  
   - 실제 MIDI 피치와 **완전 일치**보다, **청감상 큰 차이가 없는 수준**에서 프렛·줄(노드)을 잡는다 (휴리스틱·편곡 허용).

4. **템포**  
   - **MIDI에서 정의된 템포**를 악보에 사용한다 (풀믹스만으로 템포를 덮어쓰지 않는다).

5. **온셋·표기**  
   - 악보의 노드는 **온셋의 흐름·연속**으로 표현한다.  
   - **같은 시각에 여러 온셋**이 겹치면 **괄호 `()`** 로 묶어 표시한다.  
   - **지속(붉은 바로 이어지는 긴 음)** 은 **온셋만** 타브에 반영하고, 길이만으로 늘어난 “뒤꼬리”는 **노드로 반복 표기하지 않는다**.  
   - **한 마디 안에서 같은 음을 연속으로 연주**해 온셋이 여러 번 찍히면, 그 온셋들은 **모두 표현**하여 실제 연주와 맞춘다.

## 6. 사용자 제안에 대한 타당성 검토

### 6.1 “주로 연주되는 음만 MIDI/온셋으로 남기자” (잡음·이상 온셋 제거)

- **타당하다.** README에도 **분리 실패 시 보컬 잔류·오검출**, **Basic Pitch의 허위 온셋** 문제가 이미 정리되어 있다.
- **주의:** “무엇을 주된 음으로 볼지”를 알고리즘으로 정의해야 한다. 단순 임계값만으로는 **스트러밍·약한 온셋**이 날아갈 수 있다.
- 구현 시 고려할 **후보 기준** (코드에 넣을 때는 별도 모듈·플래그로):  
  - MIDI **velocity / confidence** 임계값 (모델이 제공할 때)  
  - **최소 지속 시간** 필터 (클릭성 짧은 스파이크 제거)  
  - **피치 클래스별** 빈도·에너지 기반 이상치 제거  
  - (선택) 추정 **코드 진행**과의 **하모닉 정합성** — 단, 과도하면 실제 지나는 음까지 지워질 수 있음

이 문서는 **필터를 도입한다**는 방향만 고정하고, 구체 수치는 **실험·평가**로 정한다.

## 7. 이 문서에 추가로 두는 프로젝트 사실 (README·코드 기준)

- **탭 렌더 모드:** `TAB_RENDER_MODE=transcription | arrangement` — 기본 미설정은 transcription 계열. 편곡·코드 중심은 `arrangement`.  
- **품질 게이트:** `TAB_ARRANGEMENT_MIN_RECALL` 등 — 변경 시 **의도적 튜닝**임을 커밋/설명에 남긴다.  
- **환경:** 백엔드는 **ffmpeg** PATH 필요, 프론트는 **alphaTab vendor 빌드**(`npm install` postinstall) 등 — AI가 새 의존성을 추가할 때는 **README와 충돌 여부**를 확인한다.  
- **한계:** 동일 MIDI에 대해 **여러 프렛보드 해가 존재**한다 — “정답 타브”는 유일하지 않을 수 있음을 문서·테스트 기대치에 반영한다.

---

## 8. 비목표 (AI가 임의로 넣지 말 것)

- 유튜브 외 **모든 스트리밍 서비스**를 한 번에 완벽 지원한다는 전제로 대규모 확장.  
- **저작권 없는 곡** 외 법적 이슈를 코드로 “해결”한다는 취지의 기능.  
- 사용자 요청 없이 **완전히 다른 악기**를 기본 타깃으로 바꾸는 것.  
- 프로젝트 전체를 **다른 프레임워크로 포팅** (명시 요청 시에만).

## 9. TDD를 이 프로젝트에 적용하려면

### 9.1 무엇이 “TDD하기 좋은” 부분인가

- **순수 규칙:** 온셋 병합, 마디 그리드 스냅, 괄호 그룹핑, 카포 후보 스코어링, AlphaTex 문자열 조립 규칙 등 **입력·출력이 명확한 함수**.  
- **계약:** “MIDI 이벤트 리스트 → 필터링된 이벤트 리스트” 같은 **단계별 변환**.

**전사 모델·Demucs·yt-dlp** 자체는 E2E/스모크·픽스처 기반으로 두고, **단위 테스트의 1순위는 도메인 규칙**이다.

### 9.2 사용자가 지정해 주어야 하는 것

1. **우선 검증할 유스케이스 1~3개** (예: “겹침 온셋 → 괄호”, “연속 동일 음 온셋 전부 유지”).  
2. **테스트 입력 형식** — 짧은 MIDI 또는 이벤트 배열 JSON 등 **고정 픽스처**를 허용할지.  
3. **품질 기준 한 줄** — 예: “필터 후에도 스트러밍 누락이 없어야 한다” vs “고스트 노트만 줄인다” 등 **트레이드오프 방향**.  
4. **레거시 동작 보존** — `transcription` vs `arrangement` 중 어떤 모드가 **회귀 테스트 대상**인지.

### 9.3 개발 루프 (요약)

1. 실패하는 테스트 작성 (행동·예시 데이터 명시).  
2. 최소 구현으로 통과.  
3. 리팩터 — 이때 **도메인 용어(노드, 온셋, 음원)** 를 테스트 이름에 쓴다.  
4. 파이프라인이 커지면 **Repository/어댑터**로 I/O를 분리하고, 도메인은 **순수 함수**에 가깝게 유지한다.
