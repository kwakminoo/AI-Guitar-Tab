## 1. 문제 정의서

### 해결할 문제

- **입력:** 혼합 믹스(보컬·드럼·베이스 등이 함께 있는) 또는 단일 트랙에 가까운 기타 오디오(주로 WAV).
- **출력:** 시간축 상의 **노트 이벤트**(시작·종료·피치/MIDI)와 이를 **기타 현·프렛**으로 사상한 결과, 그리고 악보/뷰어용 스코어(JSON 등).

핵심 난제는 다음과 같습니다.

1. **소스 분리:** 믹스에서 기타 스템을 분리해 전사 품질을 올린다.
2. **다성부(폴리포닉) 전사:** 동시에 여러 음이 울릴 수 있는 기타 연주에서 노트 단위로 겹침을 풀어 낸다.
3. **모호성 해소:** 동일 MIDI를 여러 현·프렛 조합으로 연주할 수 있어, **연주 가능한 타브**로의 매핑이 다대일 문제이다.

### 예측 목표

- **기능 목표:** 유튜브 URL 등에서 추출한 오디오에 대해, 사용자가 **편집 없이도 참고용 타브**로 쓸 수 있는 수준의 노트·리듬 정보를 자동 생성한다.
- **품질 목표(정성):** 단일 기타 위주·비밀도가 낮은 구간에서는 누락·오검출이 적고, 밀도가 높은 코드·스트러밍에서는 사람이 손볼 여지가 남는 수준을 **베이스라인**으로 둔다.
- **운영 목표:** GPU 유무·OS 환경에 따라 **ONNX/CPU 추론** 등으로 서빙 가능성을 유지하고, 전사 모듈 실패 시 **단순 대체 경로**(예: 모노포닉 가정)로 서비스 중단을 피한다.

## 2. 데이터 설명

### 출처

- **학습 데이터(본 저장소 외부):** 베이스라인 전사에 사용하는 **Basic Pitch** 계열 모델은 공개 논문/구현(Spotify, ICASSP 2022 계열 가중치)이며, 본 프로젝트 코드에서 **추가 학습(fine-tuning)** 을 전제로 하지 않은 **사전학습 가중치 추론**을 사용한다.
- **분리 모델:** **Demucs** 등 사전학습 소스 분리 모델로 믹스에서 기타 스템을 추출한다.
- **런타임 데이터:** 사용자가 지정한 **유튜브 등 URL**에서 내려받은 오디오(`yt-dlp` 등), 또는 로컬 WAV. 저장소에 고정 **공개 벤치마크 세트**는 포함하지 않는 구성일 수 있다.

### 크기·형식

- 클립 길이는 파이프라인 설정에 따라 **수십 초 단위**로 제한될 수 있다(서버 타임아웃·비용 관리).
- 샘플링 레이트·채널은 전처리 단계에서 라이브러리 기본에 맞게 처리된다(모델별 요구사항 준수).

### 주요 변수(특징·라벨 관점)

| 구분 | 내용 |
|------|------|
| **입력 특징** | 시간–주파수 도메인 오디오 파형(및 내부적으로 STFT 등으로부터 추출되는 표현) |
| **중간 표현** | 분리된 기타 스템 파형 |
| **모델 출력(전사)** | 노트 이벤트 리스트: `(시작 시각, 종료 시각, MIDI 피치, 진폭/신뢰도류)` |
| **후처리 출력** | 현 번호(1–6), 프렛, 박자·템포·조성·코드 등 악보 메타와 결합된 스코어 |

## 3. 기본 모델 실행

### 베이스라인 구성

1. **소스 분리:** Demucs 계열로 기타 스템 추출.
2. **전사:** **Basic Pitch** `predict`로 노트 이벤트 추출(ICASSP 2022 경로 등 구현체 기본 가중치).
3. **매핑:** MIDI → 표준 튜닝 기준 **현·프렛** 휴리스틱(및 키·카포 등 하모니 추정과 결합 가능).

### 실행 결과(본 저장소 기준)

- **정량 표:** 이 저장소만으로는 **고정 테스트 곡 세트에 대한 Accuracy/F1 표**가 자동 산출되도록 구성되어 있지 않을 수 있다. 재현 가능한 수치는 **별도 평가 스크립트**(참조 MIDI 대비 `mir_eval` 등)를 추가해 채운다.
- **정성 결과:** 추론 성공 시 API/서비스 경로에서 **노트 이벤트 시퀀스 → AlphaTab 등 스코어 JSON**이 생성된다. Basic Pitch는 폴리포닉 전사를 목표로 설계되어, 기타 단일 트랙에 **실용적인 1차 전사**를 제공하는 것이 일반적인 사용 시나리오다.

## 4. 성능 지표 해석

이 과제는 “곡 하나 = 하나의 클래스” 같은 단순 분류가 아니라 **시계열 구조화 예측**에 가깝다. 지표는 아래처럼 해석하는 것이 타당하다.

| 지표 | 의미(본 과제 맥락) |
|------|---------------------|
| **Accuracy** | 프레임 단위 피치 클래스가 있을 때, 시간축을 일정 간격으로 자른 **피치 정확도**. 다성부·무보음 구간에서는 클래스 불균형으로 수치만으로는 낙관적일 수 있다. |
| **Precision** | 모델이 “노트가 있다”고 한 시점(온셋) 또는 피치가 **얼마나 허위 양성이 적은지**. 스트러밍에서幽霊 노트가 적은지 평가할 때 유용하다. |
| **Recall** | 실제로 울린 노트/온셋을 **놓치지 않았는지**. 누락이 많으면 타브가 비어 보인다. |
| **F1-score** | Precision과 Recall의 조화 평균. 온셋·피치를 함께 맞춘 **이벤트 단위 F1**이 자주 쓰인다. |
| **시간 오차** | 온셋/오프셋이 참값 대비 **몇 ms/ms 단위로 어긋나는지**(리듬 체감에 직결). |

참조 악보(MIDI/GuitarPro 등)가 있을 때는 **mir_eval** 스타일의 온셋·피치 매칭으로 위 지표를 산출하는 것이 표준에 가깝다.

## 5. 현재 한계 정리

### 모델·데이터 측면

- **도메인 갭:** Basic Pitch는 범용 폴리포닉 전사에 가깝고, **특정 장르·이펙터·슬라이드·하이브리드 피킹** 등 기타 전용 데이터로 재학습되지 않았다면 한계가 드러난다.
- **분리 품질:** Demucs 분리가 불완전하면 전사 입력에 **보컬 잔류·박자 왜곡**이 남아 노트 오검출이 늘어난다.
- **라벨 모호성:** 동일 음을 여러 위치에서 연주 가능해, **“정답 타브”는 유일하지 않다.** 평가 시에도 MIDI·프렛보드 제약을 어떻게 둘지 정의가 필요하다.

### 시스템·운영 측면

- **환경 의존:** Windows 등에서 ONNX/TensorFlow 런타임 이슈로 전사 모듈 import가 실패할 수 있어, 구현상 **대체 경로(예: librosa 기반 모노포닉 가정)** 로 품질이 떨어질 수 있다.
- **실시간성·비용:** 분리 + 딥 전사는 **지연·메모리** 부담이 크다. 클립 길이 제한은 품질·안정성 트레이드오프다.

### 개선 방향

1. **평가 체계:** 고정 검증 세트 + 참조 MIDI로 온셋 F1·피치 정확도·시간 오차를 **버전별로 기록**한다.
2. **도메인 적응:** 기타 솔로/스트러밍 위주 소규모 데이터로 **fine-tuning** 또는 후처리용 **작은 분류기/CRF** 를 얹는다.
3. **프렛보드 디코딩:** 물리적 연주 가능성·포지션 이동 비용을 반영한 **디코딩(동적 계획법 등)** 으로 타브 가독성을 높인다.
4. **파이프라인:** 분리 모델 교체·앙상블, 전사 전 **노이즈 감쇠** 등 전처리 실험을 체계화한다.
